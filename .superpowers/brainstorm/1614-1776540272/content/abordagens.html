3 abordagens para melhorar o fluxo de calibração

Contexto: tela grande, erros difíceis de localizar, PDF obrigatório mas só validado no envio

A

Validação inline + feedback progressivo

A tela continua como uma página única, mas cada campo valida assim que o usuário interage — PDF mostra erro imediatamente se inválido, campos obrigatórios ficam marcados em vermelho ao sair do foco, ícone de status ao lado de cada seção (✓ completo / ⚠ falta algo).

Prós

  • Menor risco de regressão
  • Usuário não perde contexto
  • Implementação mais rápida

Contras

  • Tela continua longa
  • Não resolve "por onde começo"
B

Wizard em etapas

A tela é dividida em 3 etapas sequenciais: (1) PDF — upload obrigatório, extração por IA opcional; (2) Cabeçalho — datas, responsável; (3) Campos de medição + revisão. Cada etapa valida antes de avançar. Barra de progresso no topo.

Prós

  • Resolve "por onde começo"
  • Erros sempre próximos ao campo
  • PDF validado antes de continuar

Contras

  • Refatoração maior do componente
  • Usuário avançado pode achar burocrático
C

Painel de status fixo + seções colapsáveis

Barra lateral (ou sticky header) mostra o status de conclusão em tempo real: PDF ✓, Datas ✗, Campos ⚠. As seções do formulário são colapsáveis — o usuário expande o que precisa preencher. Clicando no item do painel, navega direto para a seção com problema.

Prós

  • Visão geral sempre disponível
  • Mantém acesso a tudo ao mesmo tempo
  • Bom para usuários avançados

Contras

  • Mais complexo de implementar
  • Pode ser pesado em telas pequenas

Minha recomendação: B (Wizard) — resolve o erro de PDF antes de qualquer outra coisa e elimina a ambiguidade de "por onde começo". A refatoração do componente de 900 linhas também é necessária de qualquer forma.
